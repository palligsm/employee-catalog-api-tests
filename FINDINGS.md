# Findings — Potential Gaps in the Employee Catalog API

Observations captured while exploring the service and building the automated
tests. These are the kinds of defects/inconsistencies a tester would raise with
the development team. Severity is a rough judgement of business impact.

| # | Severity | Endpoint | Observation |
|---|----------|----------|-------------|
| 1 | **High** | `POST /employees`, `GET /employees`, `GET /employees/{id}` | **`dateOfBirth` is truncated to the year.** Sending `"1991-04-12"` on create returns `"dateOfBirth":"1991"`, and `GET /employees` **and** `GET /employees/{id}` both return just the year (`"1991"`). Day and month are silently lost, so date of birth cannot be relied upon. |
| 2 | Low | Docs vs. implementation | **Docs misrepresent the `dateOfBirth` format.** The OpenAPI example for `GET /employees/{id}` shows a full ISO date-time (`1980-03-15T00:00:00.000Z`), but the live endpoint returns only the year string (e.g. `"1991"`), consistent with #1. The documented example is misleading — a consumer coding to it would expect a parseable date-time. |
| 3 | Medium | `POST /hr/login` | **Login error contract does not match the docs.** On invalid credentials the service returns `{"error":"Invalid credentials"}`, but the OpenAPI spec documents `{"message":"Invalid credentials"}`. Consumers coding to the spec would read the wrong field. |
| 4 | Medium | `POST /hr/login` (success) | **Undocumented response field.** A successful login also returns `accountExpirationDate`, which is not described in the API documentation. |
| 5 | **Critical** | `POST /employees` validation | **Malformed input crashes the service (availability defect).** An empty `{}` body — or a body that omits the nested `contactInfo` object/keys (see #8) — returns `502`/`503` because the code throws an **unhandled error that crashes the single free-tier process**. Since one process serves both the API and the `/api-docs` Swagger UI, *both go down together* until Render auto-restarts, and re-sending the same payload re-crashes it, so a client that retries 5xx can hold the whole service down. A robust API must return a deterministic `400` and never terminate on bad input. (Distinct from #13: a missing *top-level* required field with the nested keys present returns a handled `500`, not a crash.) The suite avoids sending these crash-inducing payloads against the shared host. |
| 8 | **High** | `POST /employees` | **Documented-minimal payload crashes only when optional nested keys are ABSENT.** A body with the documented required fields plus a unique email but **omitting** `contactInfo.phone`/`contactInfo.address` crashes the host (`502`), whereas the exact same body with those optional fields **present but blank (`""`)** is accepted (`201`). So the docs are correct that only `email` is required *by value*, but the implementation cannot tolerate the *absence* of the nested keys — the same crash-on-missing-field fragility as #5, scoped to nested optional fields. The `minimalPayload_isAccepted` test now sends blank (`""`) phone/address so it exercises the accepted path; the truly-omitted-fields case remains a crash to be fixed. |
| 7 | Low | Hosting | The free-tier host cold-starts (first request up to ~60s) and occasionally returns transient `502/503`. This is an environment characteristic rather than an API defect, but it makes naive tests flaky — the suite mitigates it with warm-up + retry on login. |
| 9 | **High** | `POST /employees` validation | **`firstName` validation is ineffective — blank / whitespace names are accepted (`201`).** A whitespace-only first name (e.g. `" "` or `"  "`) with a **unique email** is created **successfully (`201`)**, producing an employee with a blank-looking name. The field is not trimmed and the stated minimum length is not enforced: a single-character name like `"A"` is also accepted (`201`), even though the only rejection message the API gives is *"First name must be at least 2 characters long."* That message is misleading — it fires **only** for a truly empty string `""` (`400`), while any value of length ≥ 1, including spaces, passes. Net effect: the "required first name" rule is trivially bypassed with a space, so records can be created with effectively no name. (For contrast, a `null` firstName crashes the host with `502` — the unhandled-input defect of #5.) Expected: reject empty/whitespace-only names after trimming, and actually enforce the documented minimum length, with a `400`. |
| 11 | Low | Auth errors (all employee endpoints) | **Inconsistent error format: auth failures return plain text, not JSON.** A missing token yields `401` with a plain-text body `Unauthorized`, and an invalid token yields `403` with plain-text `Forbidden` — whereas every other error in the API is JSON (`{"message":...}` / `{"error":...}`). A client parsing errors as JSON would fail on the auth responses. Auth errors should return the same JSON shape (e.g. `{"message":"Unauthorized"}`). |
| 12 | **High** | `POST /employees` | **Required `dateOfBirth` is not enforced — omitting it is accepted and stored as `"NaN"`.** Creating an employee without `dateOfBirth` (a documented *required* field) returns **`201`** and the record is saved with `"dateOfBirth":"NaN"`. The required-field check is missing, and the date parser writes garbage rather than rejecting the input. Expected: reject a missing `dateOfBirth` with `400`. |
| 13 | Medium | `POST /employees` | **Missing required field returns `500`, not `400`.** With the optional nested contact fields present (so #8 isn't triggered), omitting a required top-level field returns a handled but wrong status: `500` `{"message":"An error occurred","error":"User validation failed: lastName: Path \`lastName\` is required."}` (same for `contactInfo.email`). The validation is detected and reported, but surfaced as a server error instead of a client `400 Bad Request`. |
| 10 | Medium | `POST /hr/login` | **`null` password returns `500` instead of `401`.** A login with a valid username and a JSON `null` password returns **`500`** `{"error":"An error occurred during login"}` — an unhandled error rather than a clean authentication failure. (An empty-string `""`, a blank `" "`, a `null` username, an empty-string username and a wrong password are all correctly rejected with `401`; only the `null` **password** mishandles the input.) No token is issued, so it isn't a security hole, but invalid input should yield a deterministic `401`/`400`, never a `500`. Same unhandled-input theme as #5. |

## How the tests relate to these findings

- The suite asserts on **status codes and behaviour** (e.g. duplicate email → 400,
  auth required → 401, delete then 404), which is stable regardless of the
  `dateOfBirth` quirk.
- Because finding #1/#2 make `dateOfBirth` unreliable, the tests deliberately do
  **not** assert an exact stored `dateOfBirth` value; the `Employee` model keeps
  it as a raw `String` and the observation is documented here instead. This keeps
  the suite green while still surfacing the defect for the developers.
- Findings #2, #3 and #4 are documentation/contract issues; they are recorded
  here as the "gaps in the implementation" the exercise asks the tester to identify.
